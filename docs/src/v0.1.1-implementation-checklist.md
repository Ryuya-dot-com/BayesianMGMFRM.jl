## Post-v0.1.1 Implementation Outlook

The deferred work remains feasible without broadening the public model surface.
The practical order after v0.1.1 should be:

- add rank-normalized and bulk/tail diagnostic enrichment to the shared row
  vocabulary;
- add predictive-path, calibration, and category-functioning rows after that
  diagnostic schema is stable;
- add rater-homogeneity and broader interpretation-policy rows after the report
  shape is stable;
- expand validation bundles only after those schemas settle, so versioned
  exports do not churn repeatedly;
- keep broader generalized, external-validation, model-weight, and superiority
  claims blocked until their separate evidence gates pass.

## Critical Prioritization Rules

When implementation pressure forces tradeoffs, choose the work that reduces
overclaim risk first:

- fix source, transform, manifest, and report-status drift before adding new
  examples;
- standardize diagnostics before adding interpretation rows that depend on
  those diagnostics;
- add predictive-path and sensitivity labels before adding model-comparison
  summaries;
- keep DFF, model weights, sparse-superiority, free correlations, and
  exploratory loadings blocked unless their own gate is explicitly passed;
- prefer a narrower release with complete caveats over a broader release with
  ambiguous status labels.

The main anti-goal is "silent promotion." A function becoming callable is not
the same as a model surface becoming supported. Promotion requires source
alignment, design support, diagnostics, sensitivity evidence, report wording,
and release-scope rows to agree.

## Runtime-Aware Verification Policy

Julia startup, precompilation, guarded HMC smoke tests, docs rendering, and
fixture/archive regeneration are slow enough that `v0.1.1` work should use a
staged verification loop rather than running the full suite after every small
edit.

- For local code edits, first run a load or narrow smoke check such as
  `julia --project -e 'using BayesianMGMFRM'`, then the smallest script or
  test path that exercises the changed surface.
- For report, manifest, or docs changes, regenerate low-level fixtures first
  and regenerate review/archive fixtures last; otherwise stored SHA references
  will point to stale artifacts.
- After fixture regeneration, run the fixture SHA scan before the full test
  suite. A zero-mismatch scan is the fastest way to catch archive-order drift.
- Run `git diff --check` and `julia --project=docs docs/make.jl` before the
  expensive full test gate.
- Reserve `julia --project -e 'using Pkg; Pkg.test()'` for milestone slices
  and release candidates, and run it on every supported Julia version before
  tagging.

## Scope Freeze

v0.1.1 should make existing guarded generalized paths easier to review and
harder to overinterpret.

Allowed scope:

- improve status labels, manifests, reports, and release-scope rows;
- add portable reader-facing report and dossier projections while retaining
  version-1 compatibility payloads;
- strengthen rating-design, fixed-Q initialization, gauge, invariance, prior,
  pooling, and sampler reporting for the existing guarded paths;
- add explicitly approximate FACETS-compatible descriptive rows for supported
  MFRM/RSM/PCM fits;
- add reproducibility checks and runnable examples for the supported MFRM and
  both guarded generalized workflows.

Out of scope:

- exploratory MGMFRM loadings or rotation;
- free latent correlations;
- dimensionality discovery beyond a fixed confirmatory Q-matrix;
- fitted DFF model effects;
- public model weights or sparse-superiority claims;
- direct-scale generalized priors without a log-Jacobian policy;
- real-data validation or R-package overlap comparison before v0.2.0;
- rank-normalized/bulk-tail diagnostic expansion, broad predictive/category
  report schemas, rater-homogeneity contrasts, and a broader validation bundle.

## Slice 1 Definition of Done

Slice 1 is the first post-v0.1.1 dependency because later report sections need
its failure vocabulary. Treat the broader diagnostic slice as done only when
all of the following are true:

- `diagnostics`, `sampler_diagnostics`, `mcmc_diagnostics`, and
  `parameter_block_diagnostics` return compatible row vocabularies for
  `MFRMFit`, `GMFRMFit`, and `MGMFRMFit`.

  diagnostics or rank-normalized/bulk/tail diagnostics.
- Divergences, max-depth hits, E-BFMI availability, finite pointwise
  log-likelihood, and direct-constraint failures are visible at the row level.
- Diagnostics are summarized by parameter block, including rater consistency,
  loading, threshold/step, severity, and ability blocks where present.
- Report rows can represent both pass and fail states without changing schema.
- At least one guarded scalar GMFRM and one guarded fixed-Q MGMFRM smoke path
  exercise the diagnostic rows with deliberately small sampler settings.
- Public wording still labels the generalized paths as experimental after the
  diagnostic rows are added.

If these conditions cannot be met in one slice, split the work at the schema
boundary: ship shared row names and failure labels first, then add backend-
specific rank-normalized/bulk/tail upgrades in a follow-up.
